AXI DMA Architecture Guide

Chapter 1. System-Level Overview

AXI DMA의 정확한 역할 정의

주소가 있는 Memory 세계와 주소 없이 흐르는 Stream 세계를 연결하는 데이터 이동 아키텍처를 한 장씩 분해합니다.

1.1 DMA는 무엇을, 어떻게 연결하는가?

AXI DMA는 단순한 복사 회로가 아니라 CPU가 설정한 전송 조건을 받아, AXI4 메모리 버스트와 AXI-Stream 데이터 흐름 사이를 이어 주는 Bridge입니다. 이 가이드는 CPU 제어, AXI4 burst, AXI-Stream, Scatter-Gather 순서로 DMA가 어떻게 확장되는지 한 화면씩 정리합니다.

1. AXI-Lite (명령 수신용)

CPU가 DMA 제어 레지스터에 접근하는 경로입니다. 실제 payload가 아니라 source, destination, length, interrupt 같은 설정 값을 전달합니다.

2. AXI4 Full (메모리 접근용)

실제 데이터 이동 경로입니다. DMA가 AXI master가 되어 DDR 같은 system memory에 burst read/write를 수행합니다.

3. AXI-Stream (확장 경로)

가속기 IP와 직접 연결되는 데이터 파이프라인. TEA 암호화 코어, 필터, FFT처럼 주소보다 데이터 순서와 packet boundary가 중요한 블록에 붙습니다.

2.1 MMIO 실행 시퀀스

STEP 1

CONTROL 레지스터에서 Enable과 Interrupt Enable을 설정

DMA 엔진을 활성화합니다. interrupt enable을 함께 켜면 완료 시 CPU가 polling 없이 알림을 받을 수 있습니다.

STEP 2

SOURCE / DEST 레지스터에 시작 주소 기록

기본 MM2MM 구조에서는 source와 destination이 모두 메모리 주소입니다. 스트림 포트가 아니라 AXI 메모리 공간 두 곳을 연결합니다.

STEP 3

NUM 레지스터에 시작 플래그, 묶음 크기, 총 길이를 함께 기록

전송 시작 트리거입니다. Chunk는 한 번에 얼마나 큰 묶음으로 버스트를 발생시킬지, Bytes는 총 전송 길이를 뜻합니다.

STEP 4

GO가 0으로 내려올 때까지 폴링하거나 IRQ를 기다림

드라이버는 상태 레지스터를 반복해서 읽으며 완료를 확인합니다. 인터럽트 사용 시에는 pending 상태와 IRQ 출력도 함께 봅니다.

2.2 버스트 전송 설계 포인트

  • DMA 적용 목적: CPU가 큰 메모리 블록을 직접 복사하면 성능이 떨어지고 버스트 효율도 얻기 어렵습니다.
  • 필수 레지스터: source, destination, length, chunk, done/interrupt 상태가 기본입니다.
  • Chunk의 의미: chunk는 여러 burst를 어느 크기 단위로 나눠 처리할지 정하는 값이며, AXI data width와 FIFO depth 제약을 함께 받습니다.
  • 주소 제약: source와 destination의 정렬 관계가 같아야 하며, misaligned start는 내부 FSM에서 따로 처리합니다.
  • 검증 관점: 레지스터 설정 순서, 완료 상태, destination 메모리 비교가 핵심 확인 항목입니다.

2.5 AXI-Lite와 AXI4 Full의 결정적 차이

AXI-Lite (제어용)

1 주소 = 1 데이터. CPU가 DMA의 레지스터(예: Source Address)를 세팅할 때 씁니다. 한 번의 주소 전송에 딱 한 번의 데이터만 보냅니다.

AXI4 Full (데이터용)

1 주소 = N 데이터 (Burst). DMA가 메모리에서 데이터를 가져올 때 씁니다. 시작 주소 1번만 알려주면, 최대 256개의 연속된 데이터를 연달아 쏟아냅니다.

2.6 AXI4 5개 독립 채널 애니메이션 시뮬레이션

AXI4는 write address, write data, write response, read address, read data 채널을 독립적으로 운용합니다. DMA는 이 구조를 이용해 읽기와 쓰기 파이프를 분리합니다.

AXI4 memory burst가 주소 기반 전송이라면, AXI-Stream은 VALID/READY와 packet boundary만으로 흐르는 단방향 데이터 경로입니다.

3.1 Stream으로 확장하기 전 DMA 내부 구조

Read Core

SOURCE 주소를 기준으로 AR/R 채널을 발생시켜 데이터를 읽습니다. 정렬이 맞지 않으면 먼저 misaligned access를 처리한 뒤 bulk burst로 넘어갑니다.

Synchronous FIFO

읽기 측과 쓰기 측 사이의 탄력 버퍼입니다. 읽기 응답 타이밍과 쓰기 승인 타이밍이 어긋나도 바로 막히지 않도록 완충 역할을 합니다.

Write Core

FIFO에서 꺼낸 데이터를 destination 메모리에 AW/W/B 채널로 씁니다. 읽기 측과 유사하게 misaligned start와 aligned burst를 구분합니다.

구성요소 대표 상태 핵심 역할
Read Engine 대기, 비정렬 보정, 정렬 버스트, 읽기 진행 남은 바이트 수와 정렬 상태를 보고 적절한 ARLEN/ARSIZE 조합을 만든 뒤 FIFO에 밀어 넣음
Sync FIFO rooms / items 카운터 Read와 Write 사이를 decouple하여 둘 중 한쪽이 잠시 느려져도 전체가 바로 멈추지 않도록 함
Write Engine 대기, 비정렬 보정, 정렬 버스트, 쓰기 진행 FIFO에서 받은 data/strobe를 이용해 메모리 write burst를 생성하고 응답(B)을 수집함

개념도: 내부 엔진 협업 흐름

Read Engine

SOURCE 기준으로 AR/R 채널을 운용합니다. 시작 주소가 정렬되지 않았으면 첫 beat를 따로 처리하고, 이후 aligned burst로 전환합니다.

→

Sync FIFO

read 쪽에서 들어온 data와 byte strobe를 잠시 보관합니다. 양쪽 AXI 응답 속도가 달라도 DMA 전체가 바로 멈추지 않게 완충합니다.

→

Write Engine

DEST 기준으로 AW/W/B 채널을 운용합니다. FIFO에서 받은 strobe를 이용해 partial write와 마지막 beat를 안전하게 마무리합니다.

Start 조건

CSR에서 GO를 수락하면 read/write 엔진이 각각 source와 destination 주소를 기준으로 준비됩니다.

Backpressure 조건

FIFO가 가득 차면 read가 늦추고, FIFO가 비면 write가 기다립니다. 이 흐름이 양쪽 채널을 decouple합니다.

Done 조건

읽기 엔진과 쓰기 엔진이 모두 마지막 응답까지 확인했을 때 DMA 완료로 봅니다.

핵심 구현식으로 보는 DMA 엔진

RTL을 그대로 보여주기보다, 구조를 이해하는 데 필요한 식만 추려 보면 DMA core는 아래처럼 읽을 수 있습니다.

Busy 조건

read engine 또는 write engine 중 하나라도 동작 중이면 DMA는 busy입니다.

Done 조건

read 완료와 write 완료가 모두 확인되어야 전체 전송 완료입니다.

FIFO 조건

read는 FIFO room이 있을 때 밀어 넣고, write는 FIFO item이 있을 때 꺼냅니다.

read side: AR handshake 후 R beat를 FIFO로 수집합니다.
write side: FIFO에서 data/strobe를 받아 AW/W/B 순서로 destination에 기록합니다.

CPU의 역할과 Linked List의 진정한 의미

CPU가 descriptor를 만들고, DMA가 그 목록을 따라가며 흩어진 물리 메모리를 하나의 전송처럼 처리하는 구조입니다.

4.1 가장 헷갈리는 질문: BD는 어떻게 미리 메모리에 들어가 있는가?

아주 예리한 질문입니다. BD가 마법처럼 원래부터 메모리에 있는 것이 절대 아닙니다.

DMA가 움직이기 바로 직전, CPU의 디바이스 드라이버가 공유 가능한 RAM에 descriptor 변수를 만들고 값을 쓰는 것입니다. 즉, CPU가 빈 RAM을 확보하고 descriptor 내용을 한 칸씩 작성한 뒤, 그 시작 주소만 DMA에게 알려주는 과정이 먼저 일어납니다.

>_ CPU Device Driver - Pseudo C Flow

// 1. 메모리 공간 할당: CPU가 RAM 빈 공간을 확보

BD* bd0 = dma_alloc_coherent(sizeof(BD));

BD* bd1 = dma_alloc_coherent(sizeof(BD));

// 2. CPU가 직접 RAM에 값을 기록: 작업 지시서 작성

bd0->NXTDESC = bd1;// 다음 지시서는 bd1 위치

bd0->BUF_ADDR = block_A;// 첫 데이터 블록

bd0->LENGTH = 4096;

bd1->NXTDESC = TAIL_ADDRESS;// 여기까지 처리하면 종료

bd1->BUF_ADDR = block_B;

bd1->LENGTH = 2048;

// 3. AXI-Lite로 DMA에게 시작 위치와 끝 위치를 통보

DMA_REG->CURDESC = bd0;// DMA가 처음 읽을 descriptor

DMA_REG->TAILDESC = TAIL_ADDRESS;// 마지막 주소를 쓰는 순간 DMA 시작

4.2 CPU와 DMA의 분업 3단계

운영체제 환경에서는 하나의 논리 버퍼가 여러 물리 페이지로 흩어질 수 있습니다. SG DMA는 이 조각들을 descriptor chain으로 표현하고, DMA가 각 descriptor를 읽어 실제 데이터 블록을 순서대로 전송합니다.

Step 1. Software (CPU)

작업 지시서 작성

CPU는 메모리의 빈 영역에 BD들을 만들고, 각 BD에 다음 BD 위치와 실제 데이터 블록 정보를 기록합니다. 이때 linked-list 또는 ring 구조가 됩니다.

Step 2. AXI-Lite

DMA에게 시작점 통보

CPU는 AXI-Lite로 DMA 제어 레지스터에 현재 descriptor와 tail descriptor를 씁니다. 이 짧은 register write가 하드웨어 실행의 trigger가 됩니다.

Step 3. AXI4 Full

DMA의 독자적 수행

DMA는 AXI4 Full master로 descriptor를 fetch하고, descriptor가 가리키는 데이터 블록을 전송한 뒤, 완료 상태를 기록하고 다음 BD로 이동합니다.

NUM에 total bytes와 chunk 정보를 담고 GO 의미의 start trigger를 줍니다.

D. 관찰 단계

BUSY/DONE/GO 상태 또는 interrupt pending을 확인해 완료를 판단합니다.

이 흐름은 CPU 드라이버든 BFM 검증이든 공통으로 나타나는 MMIO 패턴입니다. 즉, CSR 맵은 단순하지만 상태 전이와 정렬 제약은 내부 제어기가 책임진다고 이해하면 됩니다.

실무 체크

CHUNK는 data bus width의 배수로 잡는 것이 안전합니다. 0으로 두면 data width 단위의 기본 묶음처럼 해석할 수 있고, 너무 큰 CHUNK는 FIFO depth와 backpressure 조건을 같이 봐야 합니다.

설계 시 핵심 포인트 (패킷 끝 표시 재생성)

하드웨어 IP(TEA Core) 설계 시 가장 주의할 점은, 입력 패킷의 마지막 표시를 내부 연산 지연만큼 맞춰 지연시킨 뒤 출력 패킷 끝에서 다시 내보내는 것입니다. 이 타이밍이 어긋나면 수신부 DMA가 완료 조건을 보지 못해 시스템이 멈출 수 있습니다.

Stream 입력 수락

S_TVALID와 S_TREADY가 동시에 1인 cycle만 실제 입력으로 봅니다. 이 순간의 payload와 packet-end 정보를 함께 잡아야 합니다.

TEA 연산 지연

TEA core는 64-bit block을 여러 round로 처리하므로 입력 valid가 곧바로 출력 valid가 되지 않습니다. valid와 boundary를 같은 latency로 지연시킵니다.

Stream 출력 완료

M_TVALID와 M_TREADY가 만나는 cycle에 암호문 block이 빠져나갑니다. 마지막 block에서는 지연된 packet-end를 함께 내보냅니다.

6.3 Self-Contained Summary

AXI DMA Summary Map

아래 요약은 AXI DMA의 핵심 구성 요소를 다시 정리한 최종 맵입니다. 각 항목은 앞 장의 개념, 인터페이스, 상태, 검증 포인트를 한 번에 연결합니다.

Protocol Layer

AXI4는 주소 채널과 데이터 채널을 분리해 burst 전송을 가능하게 합니다. Read는 AR/R, Write는 AW/W/B 채널을 사용하며, 각 채널은 VALID/READY handshake로 독립적으로 흐릅니다.

Control Layer

CPU는 AXI-Lite 같은 제어 인터페이스로 CSR을 설정합니다. enable, source, destination, length, chunk, start, done, interrupt가 기본 제어 축입니다.

Data Layer

DMA 내부는 read engine, FIFO, write engine으로 나누면 이해하기 쉽습니다. FIFO는 read 응답과 write 승인 타이밍이 달라도 데이터 흐름을 안정적으로 이어주는 완충 구간입니다.

Verification Layer

검증은 reset, 정상 전송, misalignment, partial beat, timeout, interrupt clear를 확인해야 합니다. 마지막에는 destination 메모리가 source와 동일한지 비교합니다.

요약 흐름

1. 구조

CSR + AXI Read + AXI Write

2. 시작

주소, 길이, chunk, GO

3. 이동

Read → FIFO → Write

4. 완료

DONE, IRQ, 응답 확인

5. 확장

SG, Stream, 가속기
