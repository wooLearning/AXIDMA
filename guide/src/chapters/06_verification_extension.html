TEA 암호화 파이프라인 구동 시뮬레이션

SystemVerilog Testbench 환경에서 검증되는 실제 유스케이스로서, 고속 메모리 맵(Memory Map) 평문 데이터가 AXI-Stream 기반의 IP를 통과하며 암호문으로 변환되는 동적 파이프라인을 분석합니다.

6.1 테스트벤치가 검증하는 실제 시나리오

메모리 초기화 (Pattern Injection)

Source Address 스페이스에 테스트 벡터(평문 데이터)를 주입하고, Target Destination 스페이스는 Clear하여 검증 무결성을 확보합니다.

레지스터 프로그래밍 (MMIO 설정)

Bus Functional Model(BFM)을 통해 인터럽트/포인터/트랜잭션 길이 기반의 DMA 레지스터를 Write한 후 GO 플래그를 Assert합니다.

결과 검증 (Golden Model 비교)

Hardware 트랜잭션 종료 시, Host는 Destination Memory Block을 인출하여 C-Reference 모델과 Byte 단위의 Mismatch를 대조합니다.

검증 체크리스트

  • Disable(EN=0) 상태에서의 자발적 Trigger 무시 기동 확인
  • Source/Destination Boundary Alignment 차이 시 Misaligned Handler 구동 확인
  • 최소 단편화(Base Beat) 및 잔여 Payload(Partial Transfer) 정합성 확인
  • TLAST Propagation 시퀀스와 연계된 종단 처리 검증
  • FIFO Capacity 한계(Full/Empty)에 따른 Backpressure 반응성 점검

순수 Memory-to-Memory 연결망 사이에 AXI-Stream 기반 Custom Accelerator를 삽입한 아키텍처입니다. TEA(Tiny Encryption Algorithm) Core는 AXI-Stream Slave 인터페이스로 파이프라인의 입력을 수신하여 64-bit 단위 비대칭 연산을 수행한 뒤, 동기화된 M_AXIS 경로를 통해 Write Engine 측으로 배출합니다.

TLAST Latency Alignment: 입력 종단 신호를 출력 데이터에 다시 맞추기

암호화 코어가 내부에서 여러 cycle을 소비하면, 데이터만 지연되는 것이 아니라 TLAST도 같은 깊이만큼 지연되어야 합니다. 아래 그림처럼 입력 D3에 붙어 있던 TLAST는 출력 C3이 나가는 순간에 맞춰 다시 발생해야 S2MM이 패킷 완료를 인식합니다.

설계 시 핵심 포인트 (TLAST 동기화)

스트림 가속기(Hardware IP) 통합 설계 시의 결정적 이슈는 입력 Packet의 종단 마커(TLAST)가 파이프라인의 연산 Latency만큼 Delay Line을 거쳐 출력 패킷의 정위치에 복원되어야 한다는 점입니다. TLAST의 Propagation 지연이 어긋날 경우 수신단(Write Engine)의 상태 전이 조건이 누락되어 Deadlock을 초래합니다.

Stream 입력 수락

`S_TVALID && S_TREADY` 조건이 High인 Cycle에서만 유효 페이로드로 래치합니다. 이 때 TLAST 메타데이터도 페이로드와 함께 파이프라인에 캡처해야 합니다.

Internal 연산 지연

TEA의 Fiestel 결합 특성 상 다단 라운드 버퍼링이 발생합니다. TLAST 역시 동일한 Shift Register 깊이(Depth)만큼 일치된 지연(Latency) 보상을 수반해야 합니다.

Stream 출력 배출

`M_TVALID && M_TREADY` Handshake 조건에 대응하여 암호문을 방출합니다. 블록의 종단 사이클에 지연된 보상 TLAST 펄스를 일치시켜 Master 버스에 회귀합니다.

6.3 Self-Contained Summary

AXI DMA Summary Map

아래 요약은 AXI DMA의 핵심 구성 요소를 다시 정리한 최종 맵입니다. 각 항목은 앞 장의 개념, 인터페이스, 상태, 검증 포인트를 한 번에 연결합니다.

Protocol Architecture

AXI4는 어드레스 페이즈와 데이터 페이즈를 Decoupling하여 파이프라인 버스트를 지원합니다. AR/R 채널과 AW/W/B 채널 간의 비동기적 독립 Handshake 메커니즘을 핵심으로 합니다.

MMIO Control Model

호스트 OS는 AXI-Lite를 경유해 IP의 CSR을 핸들링합니다. Base Address, Length, Trigger Flag 등의 제어 평면(Control Plane)이 하드웨어의 Datapath를 점화합니다.

Internal Data Dynamics

비동기 Clock Domain 또는 트래픽 지연 편차를 흡수하는 Synchronous FSM 버퍼가 매개체로 작동하며, Read FSM과 Write FSM 간의 Elastic Queueing 효과를 보장합니다.

Verification Coverage

코너 케이스(Unaligned Access, Partial Beat Pipeline 붕괴, Backpressure에 의한 Hold-off)에 대한 동적 검증 및 Reference Model 교차 검증은 결함 없는 IP 딜리버리를 좌우합니다.

요약 흐름

1. 구조

CSR + AXI Read + AXI Write

2. 시작

주소, 길이, chunk, GO

3. 이동

Read → FIFO → Write

4. 완료

DONE, IRQ, 응답 확인

5. 확장

SG, Stream, 가속기
