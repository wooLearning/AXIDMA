Chapter 7. OS Integration & Debugging

실전 통합: 주소, 캐시, SMMU, 디버깅 체크리스트

RTL과 IP 설정이 맞아도 Linux/SoC 환경에서는 주소 변환, 캐시 일관성, Stream 종료 조건 때문에 DMA가 멈춘 것처럼 보일 수 있습니다.

7.1 Virtual Address, Physical Address, DMA Address

사용자 프로그램의 포인터는 대부분 가상 주소입니다. 하지만 AXI 버스에 연결된 DMA 하드웨어는 그 값을 직접 이해하지 못합니다. DMA 레지스터와 descriptor에는 OS가 매핑해 준 DMA address 또는 SoC 물리 주소 체계에서 유효한 주소가 들어가야 합니다.

CPU Virtual

C 포인터, kernel/user가 읽고 쓰는 주소입니다.

Physical

DDR 컨트롤러 관점의 실제 메모리 위치입니다.

DMA / Bus

DMA Master가 AXI에 내보내야 하는 주소입니다.

가장 위험한 실수

`malloc()` 또는 user pointer 값을 SOURCE/DEST/CURDESC 레지스터에 그대로 쓰면 안 됩니다. Linux에서는 `dma_alloc_coherent`, `dma_map_single` 같은 DMA API로 얻은 주소를 넘겨야 합니다.

7.2 Simple Mode와 SG Mode 선택

Simple / Direct Register Mode

주소와 길이를 레지스터에 직접 쓰고 한 번의 연속 버퍼를 전송합니다. 구조가 단순하지만 큰 파일, 분산된 페이지, 반복 전송에는 CPU 개입이 잦아집니다.

Scatter-Gather Mode

CPU는 descriptor ring만 준비하고, DMA가 descriptor를 fetch/decode/update하며 여러 버퍼 조각을 이어서 처리합니다. OS 환경에서는 현실적인 대용량 전송 모델입니다.

전환 기준

“하나의 contiguous buffer를 한 번 보내기”면 Simple, “여러 page/packet/frame을 끊김 없이 밀어 넣기”면 SG로 보는 것이 안전합니다.

7.3 Cache Coherency: Flush와 Invalidate

CPU는 데이터를 캐시에 보관하고, DMA는 DDR을 직접 접근하는 경우가 많습니다. 따라서 양쪽이 같은 버퍼를 본다고 생각했는데 서로 다른 시점의 값을 보는 문제가 생깁니다.

MM2S 전

CPU가 쓴 송신 데이터가 DDR에 반영되도록 cache clean/flush를 수행합니다.

S2MM 후

DMA가 DDR에 쓴 수신 데이터를 CPU가 새로 읽도록 cache invalidate를 수행합니다.

Descriptor

SG BD는 control field와 status field를 양쪽이 번갈아 쓰므로 데이터 버퍼만큼 중요합니다.

실무 기준: `dma_alloc_coherent`는 coherent memory를 제공해 descriptor나 control buffer에 적합하고, `dma_map_single`은 일반 streaming buffer를 DMA address로 매핑하는 흐름에 자주 쓰입니다. 버퍼가 cache line을 공유하면 unrelated data까지 오염될 수 있으므로 정렬(alignment)도 함께 봐야 합니다.

Cache Sync Timeline: CPU와 DMA가 같은 데이터를 보게 만드는 순서

7.4 ACE, CHI, SMMU/IOMMU는 어디서 등장하는가?

작은 FPGA 예제에서는 DMA가 DDR 물리 주소를 직접 읽는 그림으로 충분하지만, 큰 SoC에서는 coherent interconnect와 주소 변환 장치가 DMA 경로에 들어옵니다.

ACE / ACE-Lite

CPU cache와 DMA Master 사이의 coherency를 interconnect 수준에서 일부 보장하는 AXI 확장 계열입니다.

CHI

복잡한 멀티코어 SoC에서 coherent agent들이 더 확장성 있게 통신하기 위한 프로토콜 계열입니다.

SMMU / IOMMU

장치가 내보내는 DMA address를 실제 physical address로 변환하고, 접근 권한을 제한해 잘못된 DMA 접근을 막습니다.

7.5 멈춤 현상 디버깅 순서

  1. 1. CSR 상태: Halted/Idle/Error/IOC IRQ를 먼저 확인하고, Error bit는 write-1-to-clear 정책을 구분합니다.
  2. 2. Stream handshake: ILA에서 TVALID와 TREADY가 동시에 High가 되는지, 특정 지점에서 영구 정지되는지 봅니다.
  3. 3. TLAST/TUSER: 일반 패킷은 TLAST, video stream은 TUSER(SOF)와 TLAST(EOL)의 위치를 확인합니다.
  4. 4. Address/cache: 레지스터와 descriptor에 virtual pointer가 들어가지 않았는지, flush/invalidate를 수행했는지 확인합니다.
  5. 5. FIFO depth: 연산 IP latency와 DDR stall을 견딜 만큼 FIFO/skid buffer가 있는지 검토합니다.
