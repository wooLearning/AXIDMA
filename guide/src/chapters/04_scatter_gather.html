Chapter 4. Advanced Execution Architecture

CPU의 역할과 Linked List의 진정한 의미

CPU가 descriptor를 만들고, DMA가 그 목록을 따라가며 흩어진 물리 메모리를 하나의 전송처럼 처리하는 구조입니다.

4.1 가장 헷갈리는 질문: BD는 어떻게 미리 메모리에 들어가 있는가?

Buffer Descriptor(BD)는 DMA가 알아서 만드는 것이 아닙니다. 전송을 시작하기 직전에 CPU의 디바이스 드라이버가 메모리 공간을 확보하고, 각 descriptor 안에 다음 descriptor 위치, 실제 데이터 주소, 전송 길이, 상태 필드를 직접 써 둡니다. 그 다음 CPU가 DMA의 제어 레지스터에 첫 descriptor와 tail 정보를 알려주면, DMA가 그때부터 descriptor chain을 따라갑니다.

핵심 흐름

1. Descriptor memory 확보

CPU가 DMA와 공유 가능한 RAM 영역에 BD 배열 또는 ring을 준비합니다.

2. Descriptor field 작성

각 BD에 next, buffer address, length, control/status를 채웁니다.

3. DMA register trigger

CURDESC와 TAILDESC를 설정하면 DMA가 descriptor fetch를 시작합니다.

CPU Driver Pseudo Flow

1. 메모리 공간 확보

bd0, bd1 = allocate_shared_descriptor_ring()

2. CPU가 RAM에 descriptor 작성

bd0.next = bd1, bd0.buffer = block_A, bd0.length = 4KB

bd1.next = tail, bd1.buffer = block_B, bd1.length = 2KB

3. AXI-Lite로 DMA 시작점 통보

DMA.CURDESC = bd0

DMA.TAILDESC = tail

4.2 CPU와 DMA의 분업 3단계

운영체제 환경에서는 하나의 논리 버퍼가 여러 물리 페이지로 흩어질 수 있습니다. SG DMA는 이 조각들을 descriptor chain으로 표현하고, DMA가 각 descriptor를 읽어 실제 데이터 블록을 순서대로 전송합니다.

Step 1. Software (CPU)

작업 지시서 작성

CPU는 메모리의 빈 영역에 BD들을 만들고, 각 BD에 다음 BD 위치와 실제 데이터 블록 정보를 기록합니다. 이때 linked-list 또는 ring 구조가 됩니다.

Step 2. AXI-Lite

DMA에게 시작점 통보

CPU는 AXI-Lite로 DMA 제어 레지스터에 현재 descriptor와 tail descriptor를 씁니다. 이 짧은 register write가 하드웨어 실행의 trigger가 됩니다.

Step 3. AXI4 Full

DMA의 독자적 수행

DMA는 AXI4 Full master로 descriptor를 fetch하고, descriptor가 가리키는 데이터 블록을 전송한 뒤, 완료 상태를 기록하고 다음 BD로 이동합니다.

4.3 Scatter-Gather 메모리 매핑 시뮬레이션

왼쪽의 지시서(BD) 링과 오른쪽의 실제 물리 데이터 블록(Data Chunk)의 관계에 주목하세요.
